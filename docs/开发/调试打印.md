# 调试打印

> `DEBUG_UART_ENABLE` 宏的完整行为说明。宏定义在 `include/app.h`。

---

## 开关总览

| 代码位置 | 原行为 | 宏启用时 | 原因 |
|----------|--------|---------|------|
| 电压校准 (`app_init.c`) | 从 NVR4 加载 trim | **跳过** | NVR4 读取破坏 DMA 状态，导致 printf 无输出 |
| Sleep 进入 (`app.c`) | `BLE_Power_Mode_Enter` | **跳过** | 深度睡眠断开 SWD，无法再次烧录 |
| WFI (`app.c`) | `SYS_WAIT_FOR_INTERRUPT` | **跳过** | 10.24s 广播下循环太慢，tick 无意义 |
| DIO4/5 (`app.c`, `app_process.c`) | 关闭省电 | **保持开启** | DIO5 = UART TX，关闭后 printf 无输出 |

---

## 主循环结构 (`app.c`)

```c
while (true) {
    Kernel_Schedule();          // BLE 事件处理
    Sys_Watchdog_Refresh();     // 喂狗

#ifdef DEBUG_UART_ENABLE
    // 非阻塞 tick 计数器
    static uint32_t tick_cnt = 0;
    if (++tick_cnt >= 10000) {
        tick_cnt = 0;
        PRINTF("tick\r\n");
    }
#endif

    // BLE 连接处理 ...

#ifndef DEBUG_UART_ENABLE
    BLE_Power_Mode_Enter();     // 深度睡眠
    SYS_WAIT_FOR_INTERRUPT;     // WFI
#endif
}
```

## tick 间隔估算

### 调试模式（宏启用，无睡眠）

CPU 全速运行，tick 间隔 ≈ `10000 / 循环速率`（几百 ms~几秒）。

### 低功耗模式（宏禁用，有睡眠+WFI）

CPU 只在 BLE 事件时醒来：

| 状态 | 唤醒间隔 | tick=10000 打印间隔 |
|------|----------|---------------------|
| 广播 (10.24s) | 10.24s | ~28 小时 |
| 广播 (1s) | 1s | ~2.8 小时 |
| 已连接 (500ms) | 500ms | ~1.4 小时 |

> 低功耗模式下 tick 无实用意义，因为 WFI 后迭代极慢。

## 调试流程

1. 确保 `#define DEBUG_UART_ENABLE`（`include/app.h`）
2. 编译下载（见 [BUILD.md](../BUILD.md)）
3. 串口终端 115200 8N1 → 看到 `DEVICE INITIALIZED` + 周期性 `tick`
4. 调试完毕 → 注释掉宏 → 重新编译下载 → 恢复低功耗

---

## 根因：printf 与校准冲突

`Load_Trim_Values_And_Calibrate_MANU_CALIB()` 内部通过 `Sys_GetTrim()` 读取 NVR4 Flash，会污染 DMA 控制器状态。后续 `printf_init()` 中重新配置 DMA7 通道，但 DMA 控制器全局状态已被破坏，UART TX DMA 无法触发。

**验证过程**：从可工作的初始化流程出发，逐个加回函数调用，最终定位到校准函数是唯一导致 printf 失效的调用。其他函数（`BLE_LLD_Sleep_Params_Set`、`Sys_RFFE_SetTXPower`、`Sys_Clocks_OscRCCalibratedConfig`、`Sleep_Mode_Configure`、`BLE_Is_Awake_Flag_Set`）均无影响。
