# BS300 按键功能设计文档

> 2026-07-13 | `peripheral_server_sleep` 项目
> 硬件：BUTTON_DIO = DIO2，低电平有效（按下导通 GND），弱上拉

## 1. 按键行为定义

| 操作 | 判定条件 | 行为 |
|------|---------|------|
| 短按 | 按住 < 1.5s | 音量 +1，0→1→...→9→0 循环 |
| 长按 | 按住 ≥ 1.5s | 切换模式，0→1→2→0 循环，跳过 Program 3（音频模式） |

## 3. 架构设计

### 3.1 分层模型

```
┌─────────────────────────────────────────────────────┐
│  Main_Loop (app.c)                                  │
│  ├─ button_wakeup_check()    弥补 sleep 期间丢沿     │
│  ├─ button_get_action()      检查待处理动作          │
│  ├─ bs300_set_volume_async() / bs300_switch_program_async() │
│  └─ 回调: button_done_callback() → 恢复低功耗        │
├─────────────────────────────────────────────────────┤
│  APP_Timer 200ms (app_process.c)                    │
│  └─ button_hold_tick()        递增 hold_ticks       │
├─────────────────────────────────────────────────────┤
│  DIO2_IRQHandler (button.c)                         │
│  ├─ 下降沿 → btn_pressed = true, 禁用低功耗          │
│  └─ 上升沿 → 判定短按/长按, btn_pending = 1         │
├─────────────────────────────────────────────────────┤
│  硬件层                                              │
│  ├─ DIO2: GPIO 输入 + 弱上拉                         │
│  ├─ DIO2 中断: TRANSITION 双边沿 + 去抖 DIV1024×4    │
│  └─ 唤醒源: DIO2 FALLING edge                        │
└─────────────────────────────────────────────────────┘
```

### 3.2 状态机

```
                    ┌─────────────┐
           上电     │   IDLE      │
         ─────────▶│ btn_pressed │
                   │  = false    │
                   └──────┬──────┘
                          │ DIO↓ (下降沿)
                          ▼
                   ┌─────────────┐
                   │  PRESSED    │
                   │ hold_ticks  │─── APP_Timer 200ms ──▶ hold_ticks++
                   │  递增中...   │
                   └──────┬──────┘
                          │
              ┌───────────┼───────────┐
              │ DIO↑      │           │ ticks ≥ 8
              │ (释放)     │           │ (长按判定)
              ▼           │           ▼
    ┌──────────────┐      │   ┌──────────────┐
    │ ticks < 8    │      │   │ 提前触发      │
    │ → VOLUME_UP  │      │   │ SWITCH_PROG  │
    └──────┬───────┘      │   └──────┬───────┘
           │              │          │
           └──────────────┼──────────┘
                          ▼
                   ┌─────────────┐
                   │ btn_pending │──▶ Main_Loop 处理
                   │    = 1      │──▶ 异步 I2C 发送
                   └──────┬──────┘
                          │ I2C 完成回调
                          ▼
                   ┌─────────────┐
                   │ button_done │
                   │ _callback() │──▶ 恢复低功耗
                   └─────────────┘
```

### 3.3 长按阈值

```
BTN_HOLD_THRESHOLD = 8 ticks × 200ms = 1600ms ≈ 1.5s
```

`button_hold_tick()` 由 `APP_Timer` 在 200ms 周期调用。当 `hold_ticks ≥ 8` 时**提前**触发长按动作（不等释放），用户释放后 ISR 检查 `btn_has_action` 已为 true，不再重复判定。

## 4. 关键实现细节

### 4.1 RSL10 DIO 去抖电路限制

RSL10 DIO 硬件去抖电路有已知缺陷：每个真实边沿后会产生一个**伪中断**。所有官方 SDK 示例（`ble_android_asha`、`remote_mic_rx_coex` 等）均在 ISR 中使用 `ignore_next` 跳过：

```c
void DIO2_IRQHandler(void)
{
    static uint8_t ignore_next;
    if (ignore_next) { ignore_next = 0; return; }
    // ... 处理真实边沿 ...
    ignore_next = 1;
}
```

不处理伪中断会导致：
- `hold_ticks` 被伪下降沿重置 → 长按永远无法触发
- ISR 过度触发，打乱状态机

### 4.2 Sleep 期间的按键处理

设备 sleep 时 CPU 停止，下降沿发生在中断使能之前 → ISR 收不到：

```
按下 (sleep中) → DIO2 FALLING → 唤醒 → Continue_Application → Main_Loop
                                                              │
                                              button_wakeup_check()
                                              检查 DIO_DATA 仍为低 →
                                              手动设置 btn_pressed = true
```

`button_wakeup_check()` 在 `Main_Loop` 每轮循环开始时调用，弥补 sleep 期间丢沿的问题。

**边界情况**：极短按键在 sleep 中按下+释放在 CPU 醒来前完成 → 两沿均丢失 → 按键被忽略。对于助听器使用场景，此情况极罕见且可接受。

### 4.3 低功耗控制

```
DIO↓ ISR → low_power_enable = false（禁止 sleep）
  ...
I2C 完成 → button_done_callback()
  ├─ btn_pressed == false? → low_power_enable = true
  └─ bs300_sync_is_busy()?  → 不恢复（I2C 还在进行中）
```

注意：`DEBUG_UART_ENABLE` 编译下 sleep 被 `#ifndef` 禁掉，`low_power_enable` 此时不生效。但逻辑仍正确，非 debug 编译时正常进入/退出 sleep。

### 4.4 按键动作处理（Main_Loop）

```c
if (!bs300_sync_is_busy()) {
    btn_action_t btn = button_get_action();
    if (btn == BTN_ACTION_SWITCH_PROGRAM) {
        next = (prog + 1) % 3;  // skip Program 3
        bs300_switch_program_async(next, on_btn_switch_done);
    } else if (btn == BTN_ACTION_VOLUME_UP) {
        new_vol = (vol + 1) % 10;
        bs300_set_volume_async(new_vol, on_btn_volume_done);
    }
}
```

关键保护：
- `!bs300_sync_is_busy()` 守卫：确保上一次 I2C 操作完成后再处理新动作
- Button 路径不会遇到 `bs300_switch_program_async` busy 返回 -1 的情况（因为 guards check 已过滤）
- BLE 命令路径：收到 -1 时保持 `cs_env.rx_value_changed = 1`，下次循环重试

## 5. Bug 修复记录

### 5.1 死机问题（2026-07-13）

**现象**：BLE 发送切模式/调音量或按键连续操作几次后死机。

**根因**：`bs300_switch_program_async()` 在 I2C busy 时的处理流程有 bug：

```
1. abort_requested = true       → 请求中止当前 session
2. load_struct(...)             → 加载新 Program 到 s_target
3. bs300_sync_session_init()    → memset 清零 g_bs300_sync
   ↑ 将步骤 1 的 abort_requested 清除！
   ↑ 旧 session 的 I2C 指令队列被丢弃
4. build_diff_session(...)      → 构建新指令队列
5. start_async_session(...)     → 启动新 session
   ↑ DSP 端旧 I2C 操作尚未完成，新数据覆盖上去
   → I2C / DSP 状态机混乱 → 累计数次后死锁
```

**修复**：busy 时只设 `abort_requested`，立即 `return -1`，不再覆写 session 结构体。

```c
if (bs300_sync_is_busy()) {
    g_bs300_sync.abort_requested = true;
    return -1;  // 不再继续执行后续覆写逻辑
}
// ... bs300_sync_session_init() 等仅在非 busy 时执行
```

**连带修复**：
- BLE 命令处理器：收到 -1 时 `cs_env.rx_value_changed = 1` → 下次循环自动重试
- Button action handler：已有 `!bs300_sync_is_busy()` 守卫，不受影响

### 5.2 DIO 伪中断

见 4.1 节。添加 `ignore_next` 模式和状态变迁守卫（`if (btn_pressed) return` / `if (!btn_pressed) return`）。

## 8. 开发过程踩坑记录（2026-07-13）

### 8.1 按键硬件抖动

**实测数据**（通过 DIO2 双边沿 ISR 计数）：

```
一次按下 → fall +1（偶尔 +4，严重抖动）
一次松开 → rise +2~4（每次都有抖动）
```

硬件消抖 `DIO_DEBOUNCE_SLOWCLK_DIV1024 × 4 ≈ 125us` 远远不够。机械抖动持续数十毫秒级。

**结论**：必须加软件消抖，不能仅依赖硬件消抖。

### 8.2 BLE 内核定时器不可靠

`APP_Timer` (ke_timer_set, 200ms) 在连接态下触发频率严重不足。实测 20 秒内 `button_hold_tick` 仅调用了 1 次。

**根因**：未连接时设备进入 sleep，BLE 内核定时器只在 advertising event（100ms 间隔）时检查到期。实际回调频率受 BLE 内核调度影响，不可用于需要精确计时的按键 hold 检测。

**结论**：按键 hold 计时不能依赖 BLE 内核定时器。

### 8.3 低功耗下 DIO2 ISR 不触发

**根因**：`GLOBAL_INT_DISABLE()` 在进入 `POWER_MODE_SLEEP` 前关闭了 NVIC。按键下降沿走 `WAKEUP_DIO2_FALLING` 硬件唤醒，但 NVIC 不锁存中断向量。醒来后 `GLOBAL_INT_RESTORE()` 恢复 NVIC，但 DIO2 中断已不处于 pending 态 → ISR 永远不执行。

**解决**：必须在主循环中轮询 `DIO_DATA->ALIAS[2]` 补漏。当 `low_power_enable == true`（刚从 sleep 醒）且 DIO 读到低电平时，说明是被按键唤醒的，手动设置 `low_power_enable = false` 阻止重新入眠。

### 8.4 ISR 状态机不可靠

尝试在 DIO2 ISR 中维护完整按键状态机（press/release 检测、hold 计时、消抖），结果：
- 抖动导致 `btn_hold_ticks` 不断被 `wakeup_check` 复位
- `ignore_next` 跳过伪中断的模式与真实抖动边沿交织，状态机混乱
- 多轮迭代（cooldown、bounce_recovery、action_queue）均无法稳定工作

**结论**：ISR 只做最简单的唤醒动作（设 `low_power_enable = false`），所有消抖和状态检测放到主循环轮询中。

### 8.5 最终推荐方案

```
┌──────────────────────────────────────────┐
│  ISR (DIO2_IRQHandler)                   │
│  └─ if (DIO == 0) low_power_enable=false │  ← 只管阻止休眠
├──────────────────────────────────────────┤
│  Main_Loop 轮询                          │
│  ├─ wakeup 补漏: if (lp_enable && DIO==0)│
│  │                lp_enable = false       │
│  ├─ button_tick_1ms (每~1ms)             │  ← hold 计时
│  └─ button_tick_10ms (每~10ms)           │  ← 消抖+按下/松开检测
│     └─ 连续 N 次相同值 → stable state     │
│        └─ transition → 短按/长按判定      │
├──────────────────────────────────────────┤
│  动作处理                                │
│  ├─ button_get_action() → VOLUME / SWITCH│
│  ├─ bs300_set_volume_async / switch_async│
│  └─ on_btn_done → low_power_enable=true  │  ← I2C 完成后恢复休眠
└──────────────────────────────────────────┘
```

**关键要点**：
- **消抖**：需 4+ 次连续相同读数（≥40ms）才认定状态变化
- **hold 计时**：主循环迭代计数器，不休眠时 ~1ms 精度；休眠时按钮 ISR 已阻止休眠，计时正常
- **低功耗恢复**：`on_btn_done` 回调在 I2C 同步完成后设 `low_power_enable = true`
- **WFI 控制**：`if (low_power_enable) SYS_WAIT_FOR_INTERRUPT`，按住期间自动跳过
